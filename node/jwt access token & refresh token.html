jwt 인증 흐름

출처: https://fusebit.io/blog/oauth-refresh-token-best-practices/?utm_source=www.google.com&utm_medium=referral&utm_campaign=none

  1. client에서 server(authorization server)에 인증을 요청한다.(로그인)
  2. sever(authorization server)에서 access token과 refresh token을 client에 전송한다.
  3. client는 server(resource server)의 인증이 필요한 resource에 access token을 이용해서 접근 요청을 한다.
  4. server는 client가 cookie or header에 담아서 전송한 access token을 검증해서 변조되지 않았다면 접근을 허용한다.
  5. [access token이 만료된 경우] server는 client에게 access token이 만료되었다는 응답을 하고 resource에 대한 접근을 차단한다.
  6. [access token이 만료된 경우] client는 특정 api(ex /refresh)로 refresh token을 전송한다.
  7. [access token이 만료된 경우] server는 refresh token 이 유효한지 확인을 해서 client에게 새로운 access token을 발급한다.
  8. client는 새로 발급 받은 access token을 이용해서 3번의 과정을 수행한다.

access token과 refresh token의 저장

client는 server로 부터 발급받은 access token과 refresh token을 저장해 두어서 인증에 사용해야 한다.

이전에 jwt token의 저장 위치에 대해 포스팅 했었는데(https://kimtahen.com/post/jwt-token%EC%9D%98-%EC%A0%80%EC%9E%A5-%EC%9C%84%EC%B9%98), 이 글에 따르면 cookie에 저장하되 HTTP Only 옵션과 Secure 옵션을 지정해주어야 한다고 했다.

access token 은 이러한 방식으로 저장하는데 그렇다면 refresh token은 어디에 저장해야 할까?? access token과 같은 방식으로 저장하면 refresh token의 존재 이유가 퇴색되는 것은 아닐까??

refresh token을 access token과 같은 방식으로 저장하되, refresh token을 저장하는 cookie에 path 옵션을 주어서 access token을 다시 발급 받는 url(/refresh)로 설정해주어서, refresh token이 특정 url로만 전송되도록 한다.

이런 방식으로 구성하면, access token은 server의 resource에 접근하는데만 전송되고, refresh token은 access token을 재발급 받을때만 server로 전송이 된다.

refresh token의 검증

서버에서는 db에 refresh token과 ip를 함께 저장해놓고, client의 refresh token을 이용한 요청을 한번 더 검증할 수 있다.(원래의 refresh token과 일치하는지) 만약 ip가 달라지는 경우 db의 refresh token을 제거하여 refresh token이 만료되지 않았더라도 access token을 재발급 할 수 없도록 구성할 수 있다. 또한 refresh token이 탈취되었다는 판단이 생길 때 마찬가지로 refresh token이 만료된 것 처럼 설정이 가능하다.
