프로젝트 개요

첫번째 블로그는 google firebase를 이용한 SPA 기술로 제작하였고, 두번째 블로그는 express.js를 backend로 하는 SSR 방식으로 제작했습니다.

앞선 두 프로젝트는 복잡성과 의존성이 높았고, 단순명료함과 가독성에 초점을 맞춘 블로그 제작을 목표로 진행하였습니다.

서버가 필요하지 않은 serverless 방식을 택하여, github pages와 같은 무료 호스팅 서비스를 이용할 수 있도록 하였습니다.

결론적으로 트리 구조로 배열된 마크다운 파일들을 정적 사이트로 렌더링 해주는 프로그램을 제작하게 되었습니다. 특히 Obsidian과 같은 마크다운 작성 프로그램을 통해 글을 작성한 뒤, 웹 사이트로 렌더링할 수 있습니다.

기술 스택

  • rust (rendering)
    • 마크다운 파싱과 렌더링을 담당합니다. 문서 검색 index를 생성합니다.
  • rust (wasm)
    • 문서 검색 index를 읽어서 client side에서의 문서 검색 기능을 구현합니다.
  • python (NLP)
    • 문서 검색에 기능 구현에 필요한 한국어 형태소 분석기 목적으로 사용하였습니다.
  • javascript
    • 클라이언트에서 웹 사이트 렌더링에 사용됩니다.

구현 사항

마크다운 파싱 및 렌더링

computer science
└── operating system
    ├── 1. Operating System Overview.md
    ├── 2. Processes and Threads.md
    ├── 3. CPU Scheduling.md
    └── overview.md

위와 같은 트리 구조로 작성된 마크다운 바이너리 파일과 디렉터리를 파싱하고 html로 렌더링합니다.

여러개의 트리 구조의 리소스들을 하나의 루트 디렉터리를 가진 트리 구조로 재편성 한 뒤, 결과물 리소스들을 재배치합니다.

모든 연결된 마크다운 문서나 바이너리 파일들을 상대 경로로 다시 렌더링 하며, 마크다운 문서가 많을 경우 트리 노드를 추가하여 pagination을 적용합니다.

검색 인덱스 생성 및 검색 기능 구현

문서 검색을 구현하기 위해, 문서 내의 단어에 대한 인덱스를 형성하고 바이너리 파일로 추출합니다. rust로 작성된 wasm을 이용하여 client에서는 이 바이너리 파일을 읽고, 빠른 속도로 검색어가 포함된 문서를 찾습니다.

영어와 달리 한국어는 조사가 단어에 붙어 있어서 단순히 띄어쓰기 만으로 인덱스를 생성하면 적당한 문서를 찾지 못하는 경우가 있습니다. 따라서 띄어쓰기 기준으로 나눈 단어 외에도, python의 NLP 라이브러리를 이용해 문서 내의 모든 명사를 찾아서 인덱스를 생성합니다.

검색어를 더 많이 포함하는 문서를 기준으로 priority를 부여하여 검색 결과를 정렬합니다.

하위 노드에 존재하는 문서 만을 검색하도록 하였습니다.

pre-fetch

url을 통해서 html 문서들을 가지고 오지만, 대부분의 무료 서비스들은 느리다는 단점이 있습니다. github pages의 경우에는 200ms 가량의 딜레이가 발생합니다.

이를 해결하기 위해 미리 문서를 가져와서 caching을 하도록 javascript로 구현하였습니다. 하위 문서들에 대해서 백그라운드로 fetch를 합니다.

다음 링크로 이동할 때, fetch가 완료되었다면 dom element를 교체하여 웹 페이지를 빠르게 로드합니다.

multithreading

빠르게 렌더링 작업을 수행하기 위해, 여러개의 thread를 사용하여 트리의 각 노드가 동시에 렌더링이 될 수 있도록 하였습니다. 특히 rust 언어를 이용하였기에 안정적인 멀티 스레드 프로그램을 작성할 수 있었습니다.

빠르게 형태소 분석을 하기 위하여, python code를 실행하는 여러개의 프로세스를 생성하고, process pool을 구성하여 렌더링 시 각 thread에서 이용할 수 있도록 하였습니다.

이 방식을 통하여, 렌더링 속도를 30% 가량 향상시켰습니다.

etc

그 외의 웹 기능들은 javascript와 tailwindcss를 이용해서 구현하였습니다.
