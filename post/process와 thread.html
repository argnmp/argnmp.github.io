process와 thread

Taehyeon Kim

·

process

program 실행에 필요한 모든 것을 저장하는 자료구조 program in execution

process는 4개의 영역을 포함한다

  • text: program code
  • data: static and global variables, 프로그램 어느 부분 실행하는 지에 관계없이 접근가능
  • stack: function call, context switch에 사용됨
  • heap: dynamic allocation

address space: mem 상의 process 영역

  • 0~MAX(32bit or 64bit에서 최대 주소 범위)로 logical하게 연속적이다.

stack은 높은 주소에서 낮은 주소로, heap은 낮은 주소에서 높은 주소로 증가한다.

  • text, data는 fixed size
  • stack, heap은 variable size

process state

크게 5개의 상태로 나뉜다.

  • new: process 생성되어 mem에 올라간 상태. 바로 cpu scheduling 대상은 아니다.
    • 보통은 바로 ready가 되지만 rtos같은 경우에는 작업 deadline 보장 위해 시작 못할 수도 있다
  • running: cpu에서 실행중인 상태, core 여러개면 running 상태인 것도 여러개
  • waiting: process가 event 발생 기다린다. 보통 io event
  • ready: cpu scheduling 대상인 상태. 하지만 cpu 할당 받지 못했다
  • terminated: process 실행 완료. os가 자원 회수하는 것을 기다리고 있는 상태

state 변화 사유

  • new -> ready: admission control
  • ready -> running: scheduler dispatch
  • running -> ready: interrupt (cpu time slice 소진)
  • running -> exit: instruction 모두 실행 완료
  • running -> waiting: io or event wait (io나 event 발생시 waiting가서 기다린다)
  • waiting -> ready: io or event completion

process control block

process의 kernel 내 자료구조

  • state, pc, cpu regs, cpu sched info, mm info, accounting info, io status info

context switching 시, pcb를 통해 현재 process의 상태를 저장하고 다른 process의 상태를 불러온다

process scheduling queues

여러개의 queue들이 존재한다.

  • job queue: 모든 process를 관리하는 queue
  • ready queue: cpu 할당 기다리는 process를 관리하는 queue
  • device queue: io device interrupt를 기다리는 process를 관리하는 queue

linked list로 pcb들을 연결한다

schedulers

long-term scheduler

  • new -> ready 대상 결정
  • control degree of multiprogramming
    • ready queue에 넣으면 현재 동작 중인 process의 수가 한개 증가.
    • process의 종료에 관여하지는 않는다.
  • infrequent (seconds, minutes)

short-term scheduler

  • ready -> running 결정
  • frequent (milliseconds), 따라서 scheduler의 동작이 매우 빨라야 한다

mid-term scheduler

  • process의 swap-out, swap-in 을 결정 (메모리 한정적이므로)

process의 특징

  • io-bound
    • many short cpu burst
    • io 기다리는데 많은 시간이 든다. e.g. 문서나 메신저 프로그램
  • cpu-bound
    • few very long cpu burst

context switch

현재 실행중인 process의 상태를 pcb에 저장하고 다른 process의 상태를 pcb에서 불러온다.

  • process의 상태가 pcb에 나타나 있다.

중요한 overhead

  • 그 자체로 발생하는 overhead + 부가적으로 success cache misses overhead
  • context switch는 매우 빈번하므로, 매우 빠르게 동작해야 한다
    • hardware support

process creation

pid로 process를 구별한다

proces가 child process를 만드는 방식으로 생성하기 때문에, tree의 형태를 이룬다.

child는 parent의 address space를 복제한다. 이후 program을 load할 수 있다.

  • fork()
  • exec()

fork() 이후 child와 parent는 같은 pc로 fork()의 return값을 받을 차례이고, os가 return값을 다르게 준다.

  • child에게는 0
  • parent에게는 child의 pid

process termination

마지막 code 실행하고, os에게 종료 요청. exit()

  • return data는 parent에 전달된다. (parent에서 wait()을 한다) parent가 child process의 실행을 종료할 수도 있다. abort()
  • cascading termination

interprocess communication (ipc)

두개의 모델이 있다.

  • shared memory: producer-consumer model (unbounded-buffer / bounded-buffer)
  • message passing: physical(shared memory, hardware bus), logical(logical properties)
    • synchronous: blocking send, blocking receive
    • asynchronous: non-blocking send, non-blocking receive

thread

일을 하는 기본 단위로, process내에서 공유하는 부분이 많기 때문에 lightweight process라고 한다. 하나의 process 내에서 각각의 코드를 실행한다.

pc, register set, stack가 thread마다 다르다. code, data, files(open files)를 공유한다.

blocking system call 또는 io wait시, 기다리지 않고 다른 일을 할 수 있다. process 여러개 만들지 않아서 자원 효율성이 좋다. overhead가 적다. process 하나일 때에 비해, 여러개의 thread로 여러 core에서 동시에 동작할 수 있다.

user thread

user-level thread는 os는 thread의 존재에 대해서 모르고 user-level에서 thread라는 개체를 이룬다.

  • kernel에서는 process 단위의 scheduling, kernel의 support가 없다.
  • kernel 도움 없이 thread 생성, 소멸 operation이 가능해서 가볍고 빠르다. kernel-thread는 항상 os에 요청을 해야한다.
  • kernel 자체가 single thread라면, user-level thread가 blocking system call 호출 시 전체 process가 blocking이 된다. (os는 thread의 존재를 모르기 때문)

e.g. POSIX Pthreads

kernel thread

kernel-thread는 os가 thread를 지원해서, thread를 만들어주고 thread 단위로 동작하는 것을 의미한다.

  • kernel에서 thread 단위 scheduling
  • blocking thread는 전체 process를 blocking하지 않는다. 즉 다른 thread가 동작할 수 있다.
  • thread를 서로 다른 processor에서 scheduling이 가능하다.

user-thread와 kernel thread의 hybird approach가 존재한다.

threading issues

fork()와 exec()시 thread가 복제되어야 하는가? thread의 종료 방식

  • asynchronous: 즉시 종료한다. 자원 해제 문제, shared data integrity 문제
  • deferred: 주기적으로 종료해도 되는지 확인후, 순차적으로 종료한다. signal handling
  • signal은 process에게 특정 이벤트가 발생했다는 것을 알려준다.
  • 그렇다면 이 signal을 어떤 thread에 전달해야 하는가? thread pool
